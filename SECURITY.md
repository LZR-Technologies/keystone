# Security Policy

Security matters to Keystone — the tool's whole purpose is to help projects start safe. If you
find a vulnerability in Keystone itself, please help us fix it responsibly.

## Reporting a vulnerability

**Do not open a public issue for a security problem.** A public report tells everyone about the
hole before there is a fix.

Instead, report it privately through GitHub's
[private vulnerability reporting](https://github.com/Plucca7/keystone/security/advisories/new)
for this repository. That opens a confidential channel with the maintainer.

Please include, as far as you can:

- what the vulnerability is and where it lives (file, command, or flow),
- how to reproduce it,
- the impact you foresee (what an attacker could do), and
- any suggested fix, if you have one.

## What to expect

- **Acknowledgement** within a few days, confirming we received the report.
- **An assessment** of severity and scope, discussed with you in the private advisory.
- **A fix** prioritized by severity; you will be kept informed of progress.
- **Credit** for the discovery once a fix ships, unless you prefer to stay anonymous.

## Supported versions

Keystone is in early development. Security fixes target the latest released version. Until the
project reaches a stable line, only the most recent version is supported.

| Version | Supported |
| ------- | --------- |
| 0.1.x   | ✓         |
